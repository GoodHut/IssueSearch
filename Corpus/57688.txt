 URL : "https://github.com/freebroccolo/vscode-redprl/issues/21" TITLE : redprl language server BODY : @jonsterling it would not be too difficult to modify this extension so that it can work as a language server. the advantage is you would get support in other editors almost for free and with basically the same feature set as with this current vscode extension. emacs has the lsp-mode https://github.com/emacs-lsp/lsp-mode package. you just have to set up a few configuration variables to tell emacs how to connect to the server and then the lsp-mode package provides all of the other functionality to handle things like completion and diagnostics messaging. for instance, there's the lsp-ocaml https://github.com/emacs-lsp/lsp-ocaml package which uses lsp-mode to connect to the ocaml language server https://github.com/freebroccolo/ocaml-language-server . you can see here https://github.com/emacs-lsp/lsp-ocaml/blob/master/lsp-ocaml.el l43-l55 that there is not much to it since the lsp-mode package does all the heavy lifting in a language independent fashion. language server support is finally pretty widespread across editors now too. there is language-server support functionality equivalent to lsp-mode for atom https://github.com/atom/atom-languageclient , code, emacs https://github.com/emacs-lsp/lsp-ocaml , oni https://github.com/bryphe/oni/wiki/language-support , sublime https://github.com/tomv564/lsp , neovim https://github.com/autozimu/languageclient-neovim , etc.
